01

Executive Summary

In the United States, pharmaceutical companies must conduct multi-phase clinical trials before the FDA approves a new medication. These trials require controlled exchange of sensitive data among three parties — the healthcare provider, the pharmaceutical company, and the FDA — while preserving patient privacy and preventing premature access to blinded results.

This project currently delivers a front-end proof-of-concept portal for the FDA, Bavaria (a pharmaceutical company), and Jane Hopkins Hospital (a healthcare provider). The app demonstrates the clinical trial lifecycle through role-selected demo portals, seeded trial and patient data, scoped UI permissions, and simulated workflow state managed entirely in the browser.

The implementation is built with Vue 3, TypeScript, Pinia, Vue Router, Tailwind CSS, and Vite. It models trial creation, staged approvals, patient enrollment, eligibility checks, appointment and dose logging, Bavaria batch submission, FDA treatment assignment, JH completion notification, final FDA disclosure, and post-trial reporting.

Project Scope

The current app supports a simulated blinded clinical trial lifecycle: trial creation, approval review, patient enrollment and eligibility verification, appointment and dose tracking, drug batch submission, FDA assignment, completion notification, final disclosure, and reporting across four role-selected demo portals.

In Scope
Portal

Jane Hopkins Hospital

Doctor and Admin role views for patient management, CSV import, appointments, dose tracking, eligibility, and FDA completion notification.

Portal

FDA Administration

Trial approval, eligibility criteria definition, eligible patient review, treatment assignment, and end-of-trial disclosure.

Portal

Bavaria Pharma

Trial creation, batch submission, privacy-preserving trial monitoring, archive review, and post-disclosure reports.

Implementation

Vue + Pinia Demo App

Single-page Vite application with seeded data, Pinia stores, Vue Router routes, reusable components, and Vitest/Playwright coverage.

Feature

Seeded Trial Data

Multiple synthetic trial states, synthetic patient records, ICD-10 eligibility criteria, assignment maps, and 5-dose progress tracking.

Feature

Role-Scoped UI Access

Portal-specific tabs, actions, and display fields demonstrate how each persona sees only the data needed for their workflow.

Out of Scope
Real patient data Full FDA regulatory compliance Phase 1 / Phase 2 trials Mobile native apps Payment or billing systems EHR integration Production deployment Backend API persistence

05

Non-Functional Requirements & Security

Non-Functional Requirements
NFR1

Responsiveness & Performance

UI must be responsive across device sizes. Demo state updates should reflect immediately in the active workspace.

NFR2

Usability

Interfaces must be intuitive for non-technical users. Clear navigation per portal.

NFR3

Accessibility

Meets baseline accessibility standards including readable contrast ratios and semantic HTML.

Security Requirements
S1
PII Confidentiality

Patient PII (name, DOB, address, insurance number, etc.) is hidden from FDA and Bavaria views in the demo UI. Jane Hopkins roles retain full patient views for clinical workflows.

S2
Blinded Treatment Assignment

Jane Hopkins (Doctor and Admin) must not be able to determine whether a patient is receiving the Bavaria drug or the placebo until the FDA discloses assignments at trial end.

S3
Role-Based Access Control

Jane Hopkins Doctor, Jane Hopkins Admin, FDA Admin, and Bavaria Admin profiles have distinct tab visibility and action availability through the app's access helper logic.

S4
Demo-State Guardrails

Trial actions are gated by workflow state: rejected trials stop progression, complete/rejected trials can be archived, FDA disclosure waits for assignments, notification, and dose completion.

System Architecture

The current implementation is a single-page Vue application. The login screen selects a portal role, Vue Router moves users into the trial workspace, Pinia stores hold seeded trial/patient/assignment state, and composables enforce role-specific tab visibility, workflow actions, privacy views, eligibility checks, reports, appointments, and dose progress.

Technology Stack

FrontendVue 3 + TypeScriptComposition API single-file components for login, workspace tabs, modals, data tables, forms, and reusable UI primitives.
RoutingVue RouterRoutes the demo from the portal login page to the trial workspace.
StatePiniaHolds auth, selected portal, trial state, patients, assignments, UI modals, toasts, pagination, archive filters, and workflow transitions.
StylingTailwind CSSUtility-first CSS with custom theme values for FDA, Jane Hopkins, and Bavaria visual roles.
Build ToolViteDevelopment server and production build pipeline for the Vue application.
TestingVitest + PlaywrightUnit tests cover stores, access rules, reports, appointments, dose tracking, CSV behavior, and modals; Playwright covers representative portal flows.
DataSeed ModulesIn-memory seed files provide portals, trials, patients, ICD codes, and assignments for the proof-of-concept demo.

Assumptions & Constraints

Project Assumptions
Trial Scope

Clinical Trial Demo

The app models antiviral trial workflows and related approval, dosing, assignment, and disclosure states.

Simplification

Front-End Simulation

State is held in Pinia and seeded modules; refreshes reset demo changes unless persistence is added later.

Scale

Seeded Records

The workspace includes multiple sample trials and synthetic patient records to exercise status, privacy, and workflow views.

Dose Tracking

Configurable Dose Count

Most trial flows use a 5-dose target, while created trials can specify a dose count in the trial modal.

Team Constraints
All members new to Vue 3 All members new to Tailwind CSS No backend API in current implementation No production database persistence Demo login does not validate credentials Trial and error learning approach Proof-of-concept only — not production-ready
